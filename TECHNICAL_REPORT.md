## 1. 项目概述

李尔是一套以 Python 编写的中文编程式聊天机器人。系统不依赖第三方自然语言处理框架，主要通过人工知识库、对话树、正则规则、混合相似度检索、有限状态和确定性工具完成对话。

“双鱼座”是李尔的前辈，是作者更早开发的聊天系统。李尔沿用了其部分代码、知识库和设计理念，并在此基础上进行后续重构和扩展；二者不是同一版本的简单改名关系。

它并非单一的关键词问答脚本，而是由以下子系统组合而成：

- 中文输入预处理与分词
- 问题候选召回和精排
- 对话树与多轮上下文
- 用户姓名、身份和偏好记忆
- 机器人情绪与角色风格
- 数学、翻译、日期、节日等工具
- 知识图谱与规则推理
- 主动学习和本地知识保存
- 终端、Web 和 Android 会话接口

李尔的主要价值不在于某一个全新的算法，而在于将大量传统对话技术组合成一套具有长期状态、人物性格和实际工具能力的中文系统。

## 3. 总体架构

### 3.1 交互层：`LearChatEngine`

`LearChatEngine` 负责最外层的会话生命周期，包括：

- 输入规范化
- 空消息和重复输入处理
- 回复分段发送
- 提醒、小游戏等高优先级状态
- 终端与 Web 环境差异
- 资源初始化与关闭

它相当于整个系统的应用层控制器。

### 3.2 核心路由层：`AdvancedChineseBot`

`AdvancedChineseBot.get_response()` 是核心回复流水线。它依次协调：

1. 模糊问题确认。
2. 编程报错诊断。
3. 主动学习的下一轮答案。
4. 翻译等高优先级工具。
5. 游戏和数学挑战状态。
6. 用户姓名、偏好和身份。
7. 情绪和角色状态。
8. 对话树、知识库与相似度匹配。
9. 复合句拆分。
10. 知识图谱、规则推理和通用兜底。

这种瀑布式路由直观、容易添加新功能，但随着模块增多，优先级冲突会快速增加。

### 3.3 检索层

检索层主要由 `InvertedIndex` 和 `SimilarityCalculator` 构成。

#### BM25 字符倒排索引

- 对问题文本建立字符 n-gram 索引。
- 预计算 BM25 固定贡献和文档归一化数据。
- 对常见停用字符组合进行降权。
- 使用语料指纹、文件状态、参数和缓存版本校验缓存。
- 通过临时文件与原子替换保存索引，降低缓存损坏风险。

#### 混合相似度

候选排序综合使用：

- 完全一致查找
- BM25
- 单字倒排召回
- 编辑距离
- 位置 n-gram
- Jaccard
- 纯 Python 字向量语义相似度
- 候选集合内精排

检索系统只索引问题，不索引回答。回答只有在问题组确定后才从回答池中选择，这是合理且重要的设计。

### 3.4 对话树

对话树支持：

- 一组同义问法
- 多个候选回答
- 子分支
- 局部屏蔽词
- 当前根节点记忆
- 从当前分支和根节点分别匹配

它使李尔能够在部分场景中进行比普通 FAQ 更连贯的多轮交互。

### 3.5 用户画像与记忆

系统区分：

- 永久姓名与当前会话姓名
- 永久偏好与临时偏好
- 年龄、性别等基础信息
- 爱好、喜欢、讨厌、身份和性格
- 主动遗忘请求

只有身份被确认的用户才会写入永久区域，临时访客的信息不会直接污染长期画像。这一设计比简单地把所有陈述写进同一个 JSON 更稳妥。

### 3.6 情绪与角色系统

情绪系统维护用户情绪和机器人情绪，并可以：

- 添加安慰前缀
- 添加机器人情绪自白
- 根据当前心情附加 emoji 或颜文字
- 控制游戏意愿和会话结束行为

角色系统通过前缀、后缀、语气词、词语替换、特殊回答和情绪强度改变输出风格。它属于程序化人格渲染，而不是仅仅从角色回答库中随机选择一句话。

### 3.7 确定性工具

主程序包含大量工具和领域处理器，例如：

- 中文数字和四则计算
- 单位换算
- 应用题
- BMI 与近视建议
- 日期、时间差和节日
- 成语、拼音与翻译
- 生肖、星座
- 遗传学和选择题处理
- 编程错误诊断
- 随机数和密码生成
- 井字棋、猜数字、成语接龙、骰子等游戏

这些功能在覆盖范围内往往比小型生成模型更稳定，因为结果来自明确规则而非概率生成。

### 3.8 Web 与 Android 会话

项目提供 `WebChatSession` 和 `WebSessionManager`，具有：

- 每个访客独立引擎
- 会话级锁
- 最大消息数限制
- 空闲会话回收
- 最大会话数与旧会话淘汰
- 弱引用回调，减少引用环
- 幂等关闭和大对象引用释放
- 新建 Android 对话前先回收旧引擎

这表明项目已经考虑从终端脚本迁移到实际宿主环境。

## 4. 项目复杂度评估

| 维度 | 评价 | 说明 |
|---|---|---|
| 功能复杂度 | 很高 | 聊天、记忆、人格、工具、游戏和推理共同存在 |
| 算法复杂度 | 中高 | 具备候选召回、混合相似度和精排 |
| 状态复杂度 | 很高 | 多种游戏、学习、角色和用户状态并行 |
| 工程模块化 | 中等偏低 | 有大量类，但仍集中在单一主文件和主路由 |
| 可解释性 | 高 | 大部分回答路径可追踪到具体规则或数据 |
| 开放语言泛化 | 较弱 | 未编码领域依赖通用兜底，不能等同于语言模型 |
| 维护成本 | 很高 | 新增拦截可能改变其他模块的执行机会 |

### 4.1 与普通编程式机器人相比

李尔明显高于教程级关键词机器人。其优势包括：

- 不是单一字符串包含判断，而是混合检索。
- 具有对话树和多轮状态。
- 能保存用户画像。
- 能进行模糊确认和复合句回答。
- 具有独立人格和情绪反馈。
- 提供大量可直接执行的确定性工具。
- 已经具备多访客 Web 会话管理。

### 4.2 与 1B 以内语言模型相比

李尔和小型语言模型不属于同一种技术路线。

李尔更擅长：

- 已覆盖问题的稳定回答
- 结果可解释和可人工控制
- 数学、日期、单位等确定性工具
- 长期用户偏好保存
- 极低运行资源和纯 CPU 离线使用

小型语言模型更擅长：

- 未见表达的理解
- 开放式聊天
- 改写、摘要和自由生成
- 跨领域迁移
- 根据长上下文临时组合新答案

因此，李尔不能用“相当于多少参数”准确描述。它更像语言模型外部的对话编排、记忆和工具层；在人工覆盖的任务上可以比小模型稳定，在开放语言能力上则明显弱于经过大规模预训练的模型。

## 5. “还有吗”只输出表情的故障分析

### 5.1 问题表现

对话过程：

```text
用户：长寿花
李尔：第一条知识回答 + emoji

用户：还有吗
李尔：第二条补充回答

用户：还有吗
李尔：🌹
```

当前更新包中的 `bug-before-fix.jpg` 保存了修复前的实际复现画面。

![修复前只输出表情的问题](bug-before-fix.jpg)

### 5.2 根本原因

旧代码的补充回复模板如下：

```python
more_info_templates = [
    "",
    "有的，让我继续为您介绍：\n{}",
    "OK。{}",
    "好的，我再多分享一些：\n{}",
    "没问题，还有这些信息：\n{}",
    ""
]
```

执行逻辑为：

```python
template = random.choice(more_info_templates)
template.format(additional_response)
```

两个空字符串模板不包含 `{}`。当它们被随机选中时：

1. `get_additional_response()` 已经成功取得一条补充回答。
2. 该回答已经被写入“已使用回答”列表。
3. 空模板执行 `.format(additional_response)` 后仍然是空字符串。
4. 空字符串进入情绪系统。
5. 当前机器人心情为“开心”，情绪系统仍有概率追加 `🌹` 等 emoji。
6. 最终用户只看到一个表情，正文则已经被消耗。

模板共六项，其中两项为空，因此在确实还有补充内容时，正文被丢弃的理论概率约为三分之一。

这不是换行造成的，也不是知识库中的表情。换行只影响显示；真正的问题是空模板丢弃了 `{additional_response}`，情绪系统又允许修饰空正文。

### 5.3 本次修复

#### 修复一：空模板保留正文

```python
more_info_templates = [
    "{}",
    "有的，让我继续为您介绍：\n{}",
    "OK。{}",
    "好的，我再多分享一些：\n{}",
    "没问题，还有这些信息：\n{}",
    "{}"
]
```

#### 修复二：格式化后检查正文

```python
rendered = template.format(additional_response).strip()
if not rendered:
    rendered = additional_response.strip()
```

#### 修复三：情绪层不允许凭空生成 emoji

```python
if not text.strip():
    return text
```

情绪系统只能修饰已有正文，不能把空字符串变成一条单独的 emoji 消息。

#### 修复四：全局用户回复增加防御性回退

如果其他模块将空字符串传入用户可见回复流水线，系统会使用一条有正文的安全回退，而不是继续进入称呼、角色和情绪渲染。

#### 修复五：过滤空回答

补充回答池现在会排除：

- `None`
- 空字符串
- 纯空格或纯换行字符串

#### 修复六：回答耗尽后不立即清空历史

旧逻辑在回答用完时清空已使用列表。这样用户继续发送“还有吗”，下一轮又可能从第一条开始循环。

修复后会保持耗尽状态；只有用户重新提出实际主题时，正常问答流程才可以重新选择回答。

### 5.4 建议回归测试

| 用例 | 期望结果 |
|---|---|
| 回答池有三条，连续发送“还有吗” | 三条正文依次输出，允许正文后附加 emoji |
| 回答池已耗尽 | 返回“目前没有更多内容”等完整提示 |
| 回答池中含 `""`、空格或换行 | 空内容被跳过 |
| 随机模板选择无前缀形式 | 直接输出补充正文，而不是空字符串 |
| 当前情绪为开心 | 可以输出“正文 + 🌹”，不能只输出“🌹” |
| 开启角色扮演 | 角色可以修饰正文，不能以空正文触发角色前后缀 |
| 对话树回答池 | “还有吗”继续使用当前对话树的完整回答池 |
| 知识库回答池 | “还有吗”继续使用最近命中的知识库回答池 |

## 6. 当前已知工程风险

### 6.1 主路由过长

`get_response()` 超过一千行，包含大量提前返回和优先级拦截。新功能很容易抢走本应由其他模块处理的输入。

### 6.2 单文件过大

虽然代码已经使用类划分功能，但数据、工具、状态、Web 会话和 CLI 入口仍位于同一个主文件。建议逐步拆分，而不是一次性全部重写。

### 6.3 自动化测试不足

当前最需要建立的是固定测试语料，而不是继续增加功能。至少应覆盖：

- 每个高优先级工具的正例和反例
- 容易互相抢答的相似意图
- 多轮状态的开始、继续、取消和结束
- 用户偏好的保存、查询和遗忘
- 对话树与知识库之间的优先级
- 空字符串、异常输入和超长输入

### 6.4 异常处理过宽

代码中存在较多宽泛异常捕获。虽然这能避免程序立即退出，但也可能把真实错误隐藏成无响应或通用回复。

### 6.5 表达式执行安全

数学表达式模块仍应逐步从 `eval` 迁移到 AST 白名单求值，只允许数字、合法运算符和明确数学函数。

### 6.6 正则转义警告

修复版可以通过 Python 编译检查，但当前仍有 7 处无效转义 `SyntaxWarning`。它们暂未阻止运行，后续应将相关普通字符串改为原始字符串或正确双重转义。

### 6.7 知识内容质量

程序逻辑正确并不意味着知识文本必然正确。知识库需要独立的事实校验、重复清理和版本管理。

例如复现截图中的“长寿花，也称作圣诞红”属于知识文本错误，不是本次表情故障的原因。应在数据文件中单独修正。

## 7. 推荐重构路线

不建议立即完全重写。更安全的方式是按以下阶段推进。

### 第一阶段：建立回归测试

创建一份输入与预期处理器对照表，先保护现有能力。

### 第二阶段：统一意图结果格式

所有处理器返回统一对象，例如：

```python
@dataclass
class HandlerResult:
    handled: bool
    text: str = ""
    source: str = ""
    confidence: float = 0.0
    state_update: dict = field(default_factory=dict)
    allow_emotion: bool = True
    allow_roleplay: bool = True
```

这可以代替当前混合使用字符串、列表、元组、布尔值和 `None` 的情况。

### 第三阶段：建立处理器注册表

将主回复函数中的大型瀑布拆为：

```text
高优先级状态处理器
用户信息处理器
确定性工具处理器
结构化对话处理器
知识检索处理器
推理与兜底处理器
```

每个处理器声明优先级、适用条件和是否继续向后传播。

### 第四阶段：分离回复内容和风格

推荐形成固定流水线：

```text
内容生成
→ 安全检查
→ 称呼
→ 角色风格
→ 情绪修饰
→ 繁简转换
→ 输出分段
```

严重健康、心理和安全类回复可以禁止随机角色替换或娱乐化 emoji。

### 第五阶段：拆分文件

建议的长期目录：

```text
lear/
├── engine.py
├── router.py
├── retrieval/
│   ├── bm25.py
│   ├── similarity.py
│   └── vectors.py
├── dialogue/
│   ├── tree.py
│   ├── memory.py
│   └── response.py
├── profile/
│   ├── names.py
│   └── preferences.py
├── persona/
│   ├── emotion.py
│   └── roles.py
├── tools/
├── games/
├── web/
└── data/
```

拆分时应保持现有公共接口不变，逐个迁移模块并运行回归测试。

## 8. 项目特色与创新性

### 8.1 算法层面

项目主要采用已有技术组合，暂不属于提出全新机器学习算法的研究型创新。

### 8.2 系统组合层面

李尔具有较强的个人工程创新：

- 把问题检索和回答选择明确分离。
- 把用户长期画像与临时会话画像分离。
- 对模糊命中进行下一轮确认，而不是直接胡答。
- 允许用户通过对话教授新知识。
- 把人格设计为输出后处理层，而非单独回答库。
- 对回复如何在聊天界面中分段发送进行建模。
- 让机器人自身情绪影响部分交互行为。
- 将传统聊天、工具、游戏和知识推理统一在离线程序中。

### 8.3 作品辨识度

李尔具有明显的作者设计理念，不是通用框架的简单示例，也不是大型语言模型的外壳。它更接近一款长期演化的个人数字角色和中文对话实验平台。

## 9. 结论

李尔已经是一套功能复杂、交互细节丰富的高级个人编程式聊天机器人。它在人工覆盖领域具有稳定、低成本、可解释和高度可控的优势，明显超过普通关键词机器人。

当前限制主要来自工程结构，而非功能数量。下一阶段最重要的工作是：

1. 建立自动化回归测试。
2. 统一处理器返回协议。
3. 拆分主路由和单文件。
4. 建立知识数据质量检查。
5. 加强数学表达式、安全和严肃场景边界。

完成这些工作后，李尔将从大型个人脚本进一步成长为可持续维护的中文对话系统平台。
